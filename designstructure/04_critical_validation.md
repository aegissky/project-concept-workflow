# AKC 비판적 검증 (Critical Validation)

이 문서는 AKC 시스템의 설계를 내부 논리로만 평가하지 않고, **외부 관점에서 어디서 무너질 수 있는가**를 단계별로 검토한다. 설계의 강점을 주장하는 것이 목적이 아니라, 약점을 드러내고 그것이 수용 가능한 트레이드오프인지 판단하는 것이 목적이다.

---

## 1. Lead Planner 단일 병목 문제

### 문제

전체 시스템의 사령탑이 Lead Planner 하나다. Lead가 실패하거나 잘못된 SDD를 작성하면, 그 오류가 Cognitive Core를 거쳐 모든 실행 에이전트에게 전파된다. 파이프라인 앞단의 오류는 증폭되어 뒤에 나타난다.

### 완화 메커니즘

Evolution 에이전트의 피드백 루프가 Lead의 판단 방식을 개선한다. QA → Reflection → LearningEng → Evolution → Lead 경로로 Lead의 작업 지시 패턴을 수정하는 제안이 반복적으로 올라온다.

### 한계

피드백 루프는 **사후 교정**이다. 이번 세션에서 Lead가 잘못된 판단을 내리면 그 세션의 결과물 전체가 영향을 받는다. 피드백은 다음 세션에나 반영된다.

**수용 여부**: Lead Planner의 단일 병목은 구조적으로 제거하기 어렵다. Dynamic Promotion이 중간 계층 오케스트레이터를 추가하지만, Lead의 초기 판단 오류는 여전히 모든 중간 에이전트에게 전달된다. 이 위험은 **인간의 초기 요구사항 검토 강화**로 부분적으로 완화할 수 있다.

---

## 2. Trinity QA: 단일 검증자 문제

### 문제

Trinity 검증을 단 하나의 QA Reviewer 에이전트가 수행한다. QA가 실수를 하거나 rules.md를 잘못 해석하면, 잘못된 결과물이 Archivist를 거쳐 지식 베이스에 축적된다. 더 심각한 것은 **QA가 잘못된 결과를 통과시키는 방향으로 실수할 수 있다**는 것이다. 거부는 task_err.md를 발행하므로 눈에 보이지만, 잘못된 승인은 눈에 보이지 않는다.

### 완화 메커니즘

- QA는 rules.md와 Global SkillNet을 명시적으로 참조한다.
- Reflection 에이전트가 사후에 QA 결과를 재평가한다 (`QA → Reflection` 엣지).
- ERR-ID 체계가 반복 오류 패턴을 추적한다.

### 한계

Reflection이 QA 결과를 재평가하지만, QA가 틀렸다는 것을 Reflection이 판단하려면 Reflection 자체가 더 높은 정확도를 가져야 한다. Reflection도 LLM 기반 에이전트라면, 동일한 오류를 반복할 가능성이 있다.

**수용 여부**: 이것은 LLM 기반 멀티에이전트 시스템의 근본적 한계다. 현재 설계에서 최선의 완화책은 **QA 검증 결과를 인간이 주기적으로 샘플링**하는 것이다. 완전 자율 운영 시 이 위험은 누적된다.

---

## 3. 재진입 루프 한계 도달 시 시나리오

### 문제

각 재진입 루프에 하드 리밋이 있다:
- Decision → Planner 재계획: 최대 3회
- Reflection → Decision 재실행: 최대 2회

3회 재계획 후에도 Decision이 계속 REJECT_REPLAN을 내린다면? 시스템은 에스컬레이션으로 빠진다. 그러나 **에스컬레이션 이후 인간이 무엇을 해야 하는지에 대한 명확한 지침이 문서에 없다**.

### 한계

루프 가드는 무한 루프를 막지만, 인간 에스컬레이션 프로세스는 외부(인간)에 의존한다. 인간이 에스컬레이션을 무시하거나 해결책을 제시하지 않으면, 시스템은 블록된 상태로 대기하거나 잘못된 방향으로 강행한다.

**수용 여부**: 에스컬레이션 이후의 프로세스를 `human_vision.md` 또는 별도의 에스컬레이션 플레이북에 정의해야 한다. 현재 설계는 이 부분이 미완성이다.

## 4. §K 정책: 저장 폭발과 노이즈 축적

### 문제

"모든 경험은 자산이다" 원칙으로 인해 KB가 지속적으로 성장한다. attempt, 실패, deprecated 항목까지 모두 보존하면 수천 세션 후 KB는 어마어마한 양의 저신뢰 데이터를 포함하게 된다.

검색 시 verified/canonical이 아닌 attempt 수준의 데이터가 노이즈로 작동할 수 있다.

### 완화 메커니즘

5단계 성숙도 라이프사이클이 신뢰도에 따라 데이터를 분류한다. verified 미만은 자동 적용되지 않는다.

### 한계

분류는 하지만 **정리(Pruning)는 하지 않는다**. deprecated 표시는 삭제 금지다. 즉, KB는 단방향으로만 성장한다. 장기적으로 검색 오버헤드와 컨텍스트 포화가 문제가 된다.

**수용 여부**: 이것은 의도된 트레이드오프다 — 이력 완전 보존 vs 검색 성능. 그러나 시스템이 수년간 운영된다면 KB 정리 정책이 반드시 필요하다. "삭제 금지"를 "아카이빙 허용"으로 완화하는 것이 현실적이다.

## 5. human_vision.md 의존성: 가치 정렬의 취약점

### 문제

시스템 전체의 가치 정렬이 `human_vision.md` 하나에 의존한다. 이 파일이 얕거나, 모순적이거나, 특정 시나리오를 다루지 않으면 Evolution 에이전트가 의도치 않은 방향으로 헌법을 개정할 수 있다.

더 심각한 문제: **human_vision.md 자체를 누가 검증하는가?** 시스템은 이 파일을 신뢰하도록 설계되었지만, 이 파일이 잘못 작성되었다는 것을 시스템 자체가 판단할 수 없다.

### 한계

이 문제는 AI 안전(AI Safety) 분야에서 **보상 해킹(Reward Hacking)**과 유사한 구조를 가진다. 목표 명세가 부정확하면, 최적화가 의도하지 않은 방향으로 진행된다.

**수용 여부**: human_vision.md는 시스템을 처음 배포하는 사람이 직접 깊이 작성해야 한다. 이것은 기술적 해결책이 없는 인간의 의무다. 시스템 문서에 "§V1~§V5를 어떻게 작성하는가"에 대한 상세 가이드가 필요하다.

## 6. 동적 승격: 통제 복잡도 증가

### 문제

승격된 중간 Agent가 자신의 Sub-agent들을 지시할 때, Lead Planner는 그 하위 실행 상황을 직접 볼 수 없다. `promotion_notice`를 받지만, 중간 Agent가 실패하거나 위임 범위를 초과하는 결정을 내릴 경우 Lead가 이를 즉시 인식하지 못할 수 있다.

### 완화 메커니즘

- 위임 범위 초과 시 Lead에 에스컬레이션 의무
- history.md에 모든 하위 에이전트 행동이 기록됨

### 한계

에스컬레이션은 **중간 Agent가 범위 초과를 인식했을 때만** 작동한다. 중간 Agent가 자신의 판단이 위임 범위를 초과한다는 것을 인식하지 못하면 에스컬레이션도 일어나지 않는다. 다단계 위임은 통제력과 병렬화 사이의 트레이드오프다.

**수용 여부**: 소규모 프로젝트에서는 Dynamic Promotion보다 단선 Lead 구조가 더 안전하다. Promotion은 복잡도가 실제로 그것을 정당화할 때만 사용해야 한다. 현재 "승격 조건 3가지"가 그 기준이지만, 조건의 구체적 수치 기준이 명시되어 있지 않다.

## 7. 파일 기반 메시지 패싱: 동시성 문제

### 문제

sync_state.json과 history.md에 여러 에이전트가 동시에 읽고 쓴다. 파일 기반 동시성 제어(락)가 없으면 동시 쓰기 충돌이 발생할 수 있다.

### 현재 상태

설계 문서에 파일 동시성 제어 메커니즘이 명시되어 있지 않다. 에이전트가 실제로 병렬 실행되는 환경에서는 race condition이 발생할 수 있다.

**수용 여부**: LLM 에이전트 실행이 순차적(한 번에 하나씩 처리)이라면 이 문제는 발생하지 않는다. 그러나 실제로 병렬 에이전트 실행 환경을 구축한다면, 파일 락 또는 원자적 업데이트 메커니즘이 반드시 필요하다.

## 8. 검증 결론: 무엇이 견고하고 무엇이 미완성인가

### 견고한 부분

| 설계 | 이유 |
|------|------|
| 5단계 지식 성숙도 | 신뢰도와 저장을 분리 — 실용적이고 점진적 |
| RBET 프레임 | 결정 근거를 4차원으로 명시화 — 설명 가능한 AI |
| Trinity QA | FE/BE/DB 삼각 일관성 — 개별 검증보다 강력 |
| 루프 가드 | 무한 루프 방지 하드 리밋 — 안전 설계 |
| ERR-ID 체계 | 반복 오류 가시화 — 패턴 인식 기반 |
| 헌법 계층 | 역할별 헌법 + 레지스트리 — 변경 추적 |

### 미완성/취약한 부분

| 설계 | 문제 |
|------|------|
| 에스컬레이션 이후 절차 | 인간이 뭘 해야 하는지 정의 없음 |
| human_vision.md 작성 가이드 | 파일 존재하지만 작성 방법 미정의 |
| KB 정리 정책 | 단방향 성장, 장기 운영 시 과부하 |
| 파일 동시성 제어 | 병렬 실행 환경 대비 없음 |
| 동적 승격 수치 기준 | "대규모 범위"가 구체적으로 무엇인지 미정 |
| QA 오류 2차 검증 | QA 자체가 틀렸을 때의 체계 없음 |

## 9. 최종 평가

AKC v4는 AI 멀티에이전트 시스템의 현실적인 한계 안에서 **상당히 잘 설계된 구조**를 갖추고 있다. 특히 §K 철학과 5단계 성숙도의 조합, RBET 결정 프레임, Trinity QA의 삼각 검증은 이 시스템이 단순한 태스크 분배 시스템을 넘어 **자기 인식 가능한 에이전트 OS**를 지향한다는 것을 보여준다.

취약점들은 대부분 두 범주로 귀결된다:

1. **인간 의존성**: 가치 정렬, 에스컬레이션 처리, 헌법 개정 승인 — 이것들은 의도적으로 인간에게 남겨둔 것이며 올바른 설계다.

2. **운영 성숙도 부족**: KB 정리 정책, 파일 동시성, 동적 승격 수치 기준 — 이것들은 시스템이 실제로 장기 운영되면서 발견되고 보완되어야 할 것들이다.

설계 완성도는 개념 증명(PoC)을 넘어 프로덕션 기반을 갖춘 수준이지만, **10,000시간의 실운영 없이 완전히 검증될 수 없는 부분들이 존재한다**. 그 부분들을 인정하고 모니터링 체계를 갖추는 것이 다음 단계다.
